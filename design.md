# 하루한문 (daily-recall) — 설계 (Phase 1)

## 목적과 원칙 (왜) — 신규 세션 필독, 위반하는 "개선" 금지
매일 자동으로 백엔드 개발자 기술면접 질문 1개 + 관련 개념 + 모범답안을
**표준 마크다운**으로 생성해 Notion 페이지로 발행한다.
동일 마크다운을 향후(Phase 5) 영상 스크립트로 재활용한다 → 노션 종속 문법 배제.

아래 4개는 이 프로젝트의 *존재 이유*다. 구현 편의로 깨면 도구가 목적을 잃는다.

1. **1차 목적 = 개인 능동회상 규율 도구.** 청중 확보·콘텐츠 발행이 아니다.
   "발행/공유"는 본인이 매일 마주치게 하는 수단일 뿐, 조회수·폴리시 최적화 대상이 아님.
2. **능동 회상이 유일한 작동 기제 (load-bearing).** 질문/힌트를 먼저 보고 *스스로 답한 뒤*
   `---` 아래 모범답안과 대조하는 흐름이 핵심. 자동 생성된 답을 *그냥 읽으면* 우리가 거부한
   수동 소비로 회귀한다. → 템플릿의 `---` 구분선·"먼저 스스로 답하라" 안내·답안 하단 배치는
   장식이 아니라 **불변식**. 답 펼쳐두기 / 구분선 제거 / 답 상단 이동 금지.
3. **영상(P5)은 사람이 스크립트 루프에 남는다.** 완전 자동 발행 금지, 청중은 부차.
   학습은 *시청*이 아니라 개념을 30초로 압축하는 *제작*에서 발생 → 그 압축을 자동화가 대신하면
   학습가치가 사라진다. Veo류 풀오토 발행 파이프라인으로 만들지 말 것.
4. **순서 = 습관 검증(Notion) 먼저, 영상 나중.** 이유: 파이프라인 제작 자체가 보상이 되어
   정작 공부는 안 하는 실패모드 회피. P1~P4가 실제 습관으로 붙기 전엔 P5 손대지 말 것.

## 질문 소싱 전략: 하이브리드
세 가지 순수 방식의 단점을 피하기 위한 조합.

- **토픽 택소노미(백본)** — 커버리지/진행 보장. *토픽 레벨만* 큐레이션.
- **LLM 즉석 생성** — 선택된 토픽 안에서 질문+답안 생성. 무한 확장, 수기 질문작성 X.
- **이력 기반 진행·중복방지** — 같은 토픽의 과거 질문을 프롬프트에 주입 →
  "기출과 겹치지 않게, 한 단계 더 깊게" 지시.

> 순수 큐레이션 = 고갈. 순수 LLM = 품질 드리프트 + 커버리지 보장 X.
> 하이브리드는 토픽 큐레이션으로 LLM을 가둬 품질을 잡고, 생성으로 무한 확장.

## 토픽 택소노미
한국 백엔드 신입/주니어 면접 범위 기준. **기본 균등 가중** — 출제 범위는 예측 불가하므로
특정 영역으로 쏠리지 않게 한다. 가중치는 코드 상수로 분리해 조정 가능하되,
조정 용도는 "관심사 보강"이 아니라 "약점 보강"으로 한정한다.

**`키(slug)` = 정본 카테고리 키.** QAItem·history·seed_questions·taxonomy가 전부 이 slug를 쓴다.
`표시명`은 Notion Title/Category select 등 사람이 보는 표기에만 쓰며, slug→표시명 매핑은 taxonomy가 보유.

| 키(slug) | 표시명 | 하위 토픽 예시 |
|---|---|---|
| `ds_algo` | 자료구조/알고리즘 | 해시, 트리, 정렬, 시간복잡도 |
| `network` | 네트워크 | TCP/IP, HTTP(S), 로드밸런싱, 쿠키/세션, CORS |
| `os` | 운영체제 | 프로세스/스레드, 동기화, 메모리, 스케줄링 |
| `database` | 데이터베이스 | 정규화, 인덱스, 트랜잭션/격리수준, 락 |
| `java` | 언어/런타임(Java) | JVM, GC, 컬렉션, 동시성 |
| `spring` | 프레임워크(Spring) | DI/IoC, AOP, 트랜잭션, JPA |
| `web_backend` | 웹/백엔드 일반 | REST, 인증/인가, 캐싱, 멱등성, 세션/토큰 |
| `frontend` | 프론트엔드 연관 | 브라우저 렌더링, CSR/SSR, 스토리지, 보안(XSS/CSRF) |
| `infra` | 인프라/배포 | Docker, CI/CD, 클라우드 기초, 모니터링 |
| `system_design` | 시스템 디자인 | 확장성, 캐시 전략, 메시지큐 (심화) |

## 선택 로직 (MVP)
1. 카테고리 1개 선택 (**기본 균등 랜덤**, 가중치 상수는 약점 보강용으로만 조정)
2. 해당 카테고리의 과거 출제 이력 로드
3. LLM에 `(카테고리, 목표 난이도, 기출 질문 목록)` 전달 → 신규/심화 질문 + 답안 생성
4. Notion DB에 `(Date, Category, Question, Difficulty)` 페이지 생성(상태 정본)

**난이도 진행:** 카테고리별 누적 출제 횟수에 따라 난이도를 **가중 랜덤** 선택(누적↑일수록 심화 확률↑, 기초·중급도 섞어 자연 복습). 단조 상승 아님 — 영구 포화 회피.

## 마크다운 템플릿 (표준 문법만)
```markdown
# {date} · {category} · {difficulty}

## ❓ 질문
{question}

**관련 개념:** `{kw1}` · `{kw2}` · `{kw3}`

> 먼저 스스로 답해본 뒤 아래로 내려가세요.

---

## ✅ 모범답안
{핵심 답변 (2~4문장 압축)}

### 더 깊이
{원리 / 흔한 오해 / 왜 중요한가}

### 면접관 후속 질문  *(선택 — 트림 가능)*
- {follow_up_1}
- {follow_up_2}

## 🔁 복습 큐  *(선택 — 트림 가능)*
- ({n}일 전) {past_question_oneline}
```
- 관련 개념은 `concepts` 전부 렌더(템플릿의 `{kw1~3}`는 예시 수일 뿐 3개 고정 아님).
- 제목의 `{category}`는 **표시명**으로 렌더(history·QAItem에는 slug 저장).
- 이모지는 시각 앵커일 뿐 표준 마크다운(유니코드) → 빼도 무방.
- `---` 구분선 = 능동 회상 경계 + (Phase 5) 영상 "정답 공개" 컷 경계.

---

## 상태 저장
**Notion DB가 상태 정본** — 페이지 속성 `{Date, Category, Difficulty, Question, Kind}`.
- 중복방지·진행도·복습 큐가 전부 Notion DB 쿼리에서 파생(별도 상태파일 없음).
- 단일 진실원천 → 스케줄러(GH Actions) 러너가 ephemeral이어도 상태 보존. (트레이드오프: 매 실행 Notion 읽기 의존, 1일 1회라 무해.)

---

## 전달 채널 — 질문 push(슬랙) / 답 pull(노션)
원칙 #2(능동회상)의 **물리적 강제 장치**. 질문과 답을 같은 화면에 두면 `---`만 스크롤해 수동 소비로
새기 쉬우므로, 질문과 답을 **다른 채널로 분리**한다.

- **슬랙(push)**: 매일 cron이 Notion 발행 직후, **질문 + 관련개념 힌트 + "먼저 스스로 답하라" + 노션 링크 버튼**만 전송.
  **답안(`answer_core`/`answer_deep`/`follow_ups`)은 절대 싣지 않는다** — 싣는 순간 원칙 #2 붕괴. (관련 개념은 템플릿상 `---` 위 힌트라 허용.)
- **노션(pull)**: 전체 Q&A 정본. 슬랙 버튼을 눌러야 도달 → "스스로 답한 뒤 대조"를 구조적으로 강제. 상태 정본 역할도 그대로 유지.

> 트레이드오프: 슬랙은 **보조 알림**. 전송 실패해도 노션(정본)은 무사하므로 **실패를 삼키고 진행**(파이프라인 중단 X).
> 시크릿 = `SLACK_WEBHOOK_URL`(GH Secrets). 미설정 시 자동 비활성(`SEND_SLACK = bool(URL)`). CI에선 MCP 미사용 — Incoming Webhook 사용.

## 영상 연계 (Phase 5 대비, 지금은 범위 밖)
- `##` 섹션 헤더 = 영상 장면 단위.
- `---` = 컷 경계.
- 추가 메타데이터 없이 마크다운 구조만으로 스토리보드 파싱 가능하도록 유지.
